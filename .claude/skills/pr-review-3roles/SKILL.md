## 目的

PR を **3つの役割視点**で必ずレビューし、所見を **GitHub の PR コメントとして残す**（チャットに出すだけで終わらせない）。

対象 PR: $ARGUMENTS（省略時は現在のブランチに紐づく PR を `gh pr view` で特定）

---

## 必須の3視点

レビューは以下の3観点を**必ず全て**含めること。観点ごとに見出しを分け、所見には重大度マークと根拠（`file:line` 参照）を付ける。

### 🧭 プロダクトオーナー視点
- 要件・ユーザー価値・UX を満たすか
- 既存ユーザーへの影響（データ移行・**機能の後退**）。削られた機能が意図的か確認する
- エッジケース（誤操作・取り消し導線・空状態）

### 🛠 リードエンジニア視点
- アーキテクチャ・データモデル・**マイグレーションの安全性**（CLAUDE.md の本番安全原則と照合）
- コード品質・保守性・命名・重複
- **テストカバレッジ**（クロステナント認可・回帰・E2E の有無）

### 🔒 セキュリティエンジニア視点
- 認証・**マルチテナント分離**（スコープ漏れ・ID 直叩き越境）
- マスアサインメント・SQL インジェクション・秘密情報の露出
- 招待/トークンのエントロピーとライフサイクル、レート制限の有無

---

## 手順

### 1. 差分の把握
```bash
gh pr view <番号> --json title,body,files,additions,deletions,commits
gh pr diff <番号>
```
大きい PR は「ドメインモデル → サービス → コントローラ → ビュー」の順で読む。実コードを読んで判断する（推測で書かない）。

### 2. 3視点でレビュー
各視点ごとに ✅良い点 と ⚠️/🔴/🟡 指摘を整理する。重大度の目安:
- 🔴 マージ前に直すべき（導入バグ・データ整合・認可漏れ）
- 🟡 マージ後でよい改善 / フォローアップ
- ⚠️ 意思確認が必要な論点（仕様後退など）

各視点の末尾に判定（✅マージ可 / ⚠️条件付き）を付ける。

### 3. PR コメントとして投稿（必須）
まとめを Markdown で作り、**必ず** PR にコメントする:
```bash
gh pr review <番号> --comment --body-file <file>   # 総評レビューとして
# または個別論点を残す場合
gh pr comment <番号> --body-file <file>
```
最後に「判定まとめ」表（視点 / 判定）と推奨アクションを載せる。

### 4. 指摘の行き先を確定
- 本PRで直すもの → 対応（TDD）。テストが緑になってからコミット
- 次スプリント持ち越し → `scoped-issue` スキルで issue 化し、PR にも行き先サマリをコメントする

---

## 原則
- **3視点を省略しない。** 1つでも欠けたらレビュー未完了。
- **レビューは必ず PR に残す。** チャットだけで完結させない。
- 各指摘は対応先（本PR / issue番号 / 運用）まで決めて初めて完了。
