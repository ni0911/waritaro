## 目的

`git status` と `git diff` を分析し、変更を**論理的なまとまり**ごとに分けて、統一フォーマットのコミットメッセージでコミットを作成する。

引数: $ARGUMENTS

---

## コミットメッセージのフォーマット

**必ずこの形式を守ること:**

```
prefix: 簡潔なメッセージ（命令形・日本語可）

3行目以降に補足説明を記載する。
なぜこの変更が必要だったか、何を解決するかを書く。
```

- 1行目: `prefix: 簡潔なメッセージ`（50文字以内を目安）
- 2行目: 空行（必須）
- 3行目以降: 補足説明（変更の理由・背景・影響範囲など）

### 使用する prefix 一覧

| prefix | 用途 |
|--------|------|
| `feat` | 新機能の追加 |
| `fix` | バグ修正 |
| `docs` | ドキュメントのみの変更 |
| `style` | 動作に影響しない変更（フォーマット、空白、セミコロンなど） |
| `refactor` | バグ修正でも機能追加でもないコードの変更 |
| `test` | テストの追加・修正 |
| `chore` | ビルドプロセス・補助ツール・依存関係の変更 |
| `perf` | パフォーマンス改善 |
| `ci` | CI/CD 設定の変更 |
| `build` | ビルドシステムや外部依存関係の変更 |
| `revert` | 以前のコミットの取り消し |

## 手順

### 1. 変更内容の把握

以下のコマンドで変更状況を確認する:

```bash
git status
git diff --staged
git diff
```

### 2. 適切な粒度への分割

変更を以下の基準でグループ分けする:

- **1つのコミットに入れるもの**: 同じ目的・同じ機能に関する変更
- **分けるべきもの**:
  - 機能追加とバグ修正が混在している
  - 複数の独立した機能が同時に変更されている
  - ソースコードの変更とドキュメントの変更
  - 本体のロジック変更とテストの追加
  - 設定ファイルの変更とアプリケーションコードの変更

分割案をユーザーに提示し、**承認を得てからコミットを実行**する。

### 3. コミットの実行

承認後、グループごとに以下を繰り返す:

1. 対象ファイルを `git add` でステージング
2. 決定したメッセージで `git commit` を実行

コミットコマンドの例:

```bash
git add src/feature.ts src/types.ts
git commit -m "$(cat <<'EOF'
feat: ユーザー認証機能を追加

JWT を使ったトークンベースの認証を実装した。
ログイン・ログアウト・トークン更新のエンドポイントを追加する。
EOF
)"
```

---

## ユーザーへの提示フォーマット

分割案を提示する際は以下の形式で示す:

```
## コミット分割案

### コミット 1/3
対象ファイル:
- src/auth.ts
- src/types.ts

メッセージ案:
feat: ユーザー認証機能を追加

JWT を使ったトークンベースの認証を実装した。
ログイン・ログアウト・トークン更新のエンドポイントを追加する。

---

### コミット 2/3
...
```

この分割案でよいか確認し、承認を得てから実行する。

---

## 注意事項

- `git push` はユーザーが明示的に指示するまで実行しない
- `--no-verify` オプションは使わない（フックを回避しない）
- 自動生成ファイル（`dist/`、`out/`、`*.lock` など）は原則コミットに含めない（ユーザーに確認する）
- `.env` や認証情報ファイルが変更に含まれている場合は警告してコミットしない
