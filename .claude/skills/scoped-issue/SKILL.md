## 目的

「次スプリントに持ち越す」と決まったタスクを、**着手時にすぐ動ける粒度**の issue にする。

### いつ使うか
- レビューや作業で出た指摘・改善のうち、**この PR では対応せず次スプリントに回す**もの
- 「今やる」タスクには使わない（その場で対応する）

---

## 原則

1. **1 issue = 1 要件**。複数の関心事は分割する（例: 招待コードのライフサイクルとレート制限は別 issue）。
2. **スコープを閉じる**。受け入れ条件で「どうなったら完了か」を明示する。
3. **設計判断を伴うもの**（データモデル変更・ライブラリ採用等）は本文に「ADR 対象になり得る」と明記する（CLAUDE.md 規約）。
4. 相対日付は使わず絶対日付に。関連 issue / PR を相互リンクする。

---

## issue 本文テンプレート

```markdown
## 背景
（なぜ必要か。出どころ＝どの PR/レビュー由来か。現状の問題を簡潔に）

## やりたいこと
（実装の方向性。設計が未確定なら「設計を決めてから実装」と明記し、選択肢を列挙）

## 検討事項
（ADR 対象か、データ移行、関連 issue との進め方、性能/運用上の注意）

## 受け入れ条件
- [ ] 完了条件1（観測可能な形で）
- [ ] 完了条件2
- [ ] テスト（request/model/system のどれを足すか）

## 参考
（PR番号 / 関連issue / 主要ファイルパス）
```

---

## 手順

### 1. スコープを分割
出された内容を 1要件ずつに割る。混ざっていたら「これは別 issue にします」と分ける。

### 2. ラベル確認・付与
```bash
gh label list
```
種別に応じて `enhancement` / `bug` / `security` 等。無ければ `gh label create` で作る。

### 3. issue 作成
本文はファイルに書いて `--body-file` で渡す（複数なら個別ファイル）:
```bash
gh issue create --title "prefix: 簡潔なタイトル" --label <label> --body-file /tmp/issue.md
```
タイトルは `feat:` / `fix:` / `security:` / `test:` などの prefix を付ける。

### 4. 相互リンクと報告
- 関連 issue/PR を本文や `gh issue comment` でリンク
- 由来の PR があれば「行き先サマリ」を PR にコメントしてトレーサビリティを残す
- 作成した issue 番号・タイトルを一覧で報告する

---

## 完了の定義
- 各 issue が単独で着手可能（背景・やりたいこと・受け入れ条件が揃っている）
- 1 issue = 1 要件になっている
- 由来（PR/レビュー）と関連 issue がリンクされている
