# ADR 0008: payer 廃止・burden 合計による入金額算出

- **ステータス**: Accepted
- **作成日**: 2026-03-05
- **決定者**: ni0911
- **Supersedes**: [ADR 0007](0007-shared-account-settlement.md)

## 背景

ADR 0007 では「誰が実際に払ったか（payer）」を記録し、
`transfer = sum(burden) - sum(paid where payer = 自分)` で共有口座への振込額を計算していた。

しかし実際の運用を振り返ると、**すべての支払いは共有口座のクレジットカードや引き落としから行われている**。
「A が立替払いして後で精算する」というケースが存在しないため、`payer` というフィールド自体が実態に合っていなかった。

## 問題

旧モデルでは入力者に `payer` の選択を強いるが、これは運用上意味のない項目だった。

```
旧計算:
transfer_a = sum(burden_a) - sum(amount where payer == A かつ精算対象)
```

「誰が払ったか」という概念が存在しないため、この計算式は常に誤った結果を生む。

## 決定

**`payer` カラムを完全に廃止し、入金額 = burden 合計とする。**

```
新計算:
deposit_a = sum(burden_a for 精算対象アイテム)
deposit_b = sum(burden_b for 精算対象アイテム)
```

各自が共有口座に振り込む金額は、各自の負担額の合計そのものになる。

## 変更範囲

| 対象 | 変更内容 |
|------|---------|
| `sheet_items.payer` | カラム削除（マイグレーション） |
| `template_items.payer` | カラム削除（マイグレーション） |
| `SettlementService::Result` | `transfer_a/b`, `paid_a/b`, `burden_a/b` → `deposit_a`, `deposit_b`, `total_shared_amount` |
| フォーム（SheetItem・TemplateItem） | payer ラジオボタンを削除 |
| 精算ビュー | `transfer` の正負分岐を廃止。常に「→ 共有口座 に ¥XX」または「精算なし」 |
| 共有テキスト | 「○○払い」の表記を削除 |

## 検討した選択肢

### 選択肢 1: payer を残して任意入力にする

- デメリット: 意味のないフィールドが残り続ける。「入力しなくていいが表示される」という UX の混乱を招く。

### 選択肢 2: payer を廃止（採用）

- メリット: データモデルが実態と一致する。フォームが簡素化される。計算式がシンプルになり誤りが入り込む余地がない。
- デメリット: なし（payer を利用するユースケースが存在しないため）

## 新しい SettlementResult

```ruby
Result = Data.define(:deposit_a, :deposit_b, :total_shared_amount)
```

| フィールド | 意味 |
|-----------|------|
| `deposit_a` | A が共有口座に振り込む金額（= A の burden 合計） |
| `deposit_b` | B が共有口座に振り込む金額（= B の burden 合計） |
| `total_shared_amount` | 精算対象の burden 合計（= deposit_a + deposit_b） |

## 表示例

```
精算結果
──────────────────
合計精算対象       ¥122,000
みちまの負担        ¥42,000
ゆちまの負担        ¥80,000
──────────────────
みちま  → 共有口座 に ¥42,000
ゆちま  → 共有口座 に ¥80,000
```

## 結果

### ポジティブな影響

- データモデルが実際の運用（全支払いは共有口座から）と一致するようになった
- フォームから不要な選択肢（誰が払ったか）が消え、入力ステップが減った
- 計算ロジックが `sum(burden)` のみになりシンプル化・バグリスク低減

### ネガティブな影響・トレードオフ

- 将来「A が立替払いして後で精算する」運用に変わる場合は payer の再導入が必要になる
