# ADR 0004: 精算計算ロジックの設計（isSplit フラグ方式）

- **ステータス**: Accepted
- **作成日**: 2026-03-02
- **決定者**: ni0911

## 背景

同棲2人の支払いを管理する上で、「割り勘すべき費用」と「個人の私物」が混在している。
例えば家賃・食費は2人で折半すべきだが、個人の服代・趣味の支出は割り勘対象外にしたい。

また、2人の収入差などにより必ずしも 50:50 ではなく、任意の比率（例: 60:40）で負担を分けたいケースもある。

この前提のもと、毎月の精算額を計算するロジックを設計する必要があった。

## 検討した選択肢

### 選択肢 1: 全費用を按分（私物も含めて割り勘）

全ての費用を合計して `splitRatio` で按分し、それぞれの支払い実績との差額を精算する。

- メリット: 計算がシンプル
- デメリット: 個人の私物（服代・趣味費など）まで割り勘になってしまい実態に合わない。現状の手計算でも「私物は除外」という運用をしているため、この方式では解決にならない。

### 選択肢 2: 都度「自分の負担分」を入力する明細方式

費用を入力するとき「自分が負担する金額」を直接入力する。

- メリット: 柔軟性が高い
- デメリット: 毎回計算して入力する手間が増える。現状の電卓＋LINE での手計算と手間が変わらない。「ラクにする」という目的に反する。

### 選択肢 3: 固定費・変動費を別テーブルで管理

固定費テーブルと変動費テーブルを分けて管理し、それぞれ別の計算ロジックを持つ。

- メリット: 固定費の扱いを明確に分離できる
- デメリット: 画面設計・データモデルが複雑になる。実際には固定費も変動費も「割り勘かどうか」という属性が本質なので、分離するメリットが薄い。

### 選択肢 4: isSplit フラグ方式（採用）

各費用行に `isSplit: boolean` フラグを持たせ、`true` の項目のみを割り勘対象として集計する。
`splitRatio` で各人の負担比率を設定し、支払い実績との差額を精算額とする。

```
totalSplitAmount = isSplit:true の項目の合計
burden.A = totalSplitAmount × (splitRatio.A / 100)
paid.A   = isSplit:true かつ payer:A の合計
diff.A   = burden.A - paid.A
→ diff.A > 0 なら A は B に diff.A 円支払う
```

- メリット: 1項目ずつフラグをトグルするだけで柔軟に除外できる。計算ロジックがシンプルで検証しやすい。テンプレートにも同じフラグを持たせることで毎月の設定を引き継げる。
- デメリット: `isSplit:false` の項目は精算計算に含まれないため、UI 上でグレーアウトして明示する必要がある。

## 決定

**isSplit フラグ方式を採用する。**

理由: 「私物を除外して割り勘する」という要件を最もシンプルに表現できる方式であるため。
1項目ごとにフラグをトグルするだけで運用でき、電卓＋LINE での手計算より大幅に手間が減る。
計算ロジックも単純な集計のみで、バグが入りにくく検証しやすい。

## 結果

### ポジティブな影響

- 「割り勘か私物か」をフラグ1つで表現でき、実装がシンプルになった
- テンプレートにも同じフラグを持たせることで毎月の設定を自動引き継ぎできる
- `splitRatio` スライダーで比率を自由に変更でき、収入差がある2人にも対応できる

### ネガティブな影響・トレードオフ

- `isSplit:false` の項目は精算に含まれないことを UI で明示する必要がある（グレーアウト・バッジ表示で対応）
- 1項目に「Aの負担〇%・Bの負担〇%」のような個別比率は設定できない（シート全体で1つの比率を共有）

### 今後の対応

- 将来「項目ごとに個別の比率を設定したい」というニーズが出た場合は `SheetItem` に `customRatio` フィールドを追加する設計で対応できる
- 現状は比率を整数（0〜100）で管理しているが、精度が問題になる場合は小数点対応を検討する
