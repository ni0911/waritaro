# ADR 0007: 共有口座精算モデルの採用

- **ステータス**: Accepted
- **作成日**: 2026-03-02
- **決定者**: ni0911
- **Supersedes**: [ADR 0004](0004-settlement-calculation-logic.md)

## 背景

現行の精算結果は「B → A へ 12,000円」という「一方が他方に支払う」形式だった。

しかし実際の運用では、2人が**共有口座**にそれぞれ振り込み、
共有口座から家賃・光熱費・食費などを自動引き落としする構成になっている。
この運用を現行の精算モデルでは正確に表現できない。

## 問題の例

```
A の負担: 60,000円
B の負担: 40,000円
A の支払い実績: 70,000円（払い過ぎ）
B の支払い実績: 30,000円（払い不足）
```

**旧モデル（A→B 払い）:**
`B → A へ 10,000円`

この表示では「共有口座に振り込む金額」が分からない。

**新モデル（共有口座）:**
`A ← 共有口座 から 10,000円（払い過ぎ）`
`B → 共有口座 へ 10,000円`

## 検討した選択肢

### 選択肢 1: 現行モデルを維持（A→B 払い）

- メリット: 変更不要
- デメリット: 実際の振込先（共有口座）を表現できない。「共有口座に何円入れればいいか」が直感的に分からない

### 選択肢 2: 共有口座精算モデル（採用）

各人が「共有口座に振り込む金額」を算出する。

```
transfer_a = sum(burden_a) - sum(amount where payer = A かつ精算対象)
transfer_b = sum(burden_b) - sum(amount where payer = B かつ精算対象)
```

- `transfer > 0`: 共有口座に支払う
- `transfer < 0`: 共有口座から受け取る（払い過ぎ）

- メリット: 実際の振込操作に直接対応する。`transfer_a + transfer_b = 0` が恒等式として成立し、共有口座の収支が常に合う
- デメリット: 「A から B へ」という直接送金には対応できない（今回の用途ではそのケースがないため問題なし）

## 決定

**共有口座精算モデルを採用し、精算結果を `transfer_a / transfer_b` で表す。**

理由:
- 実際の振込操作（共有口座への入金）に直接対応するため、ユーザーが判断なしに操作できる
- `transfer_a + transfer_b = 0` という不変条件があり、計算の正確性を検証しやすい

## 新しい SettlementResult

```ruby
Result = Data.define(:transfer_a, :transfer_b, :total_shared_amount, :burden_a, :burden_b, :paid_a, :paid_b)
```

| フィールド | 意味 |
|-----------|------|
| `transfer_a` | A が共有口座に支払う金額（負なら受け取り） |
| `transfer_b` | B が共有口座に支払う金額（負なら受け取り） |
| `total_shared_amount` | 精算対象の合計金額 |
| `burden_a` | A の総負担額 |
| `burden_b` | B の総負担額 |
| `paid_a` | A の実際の支払い実績 |
| `paid_b` | B の実際の支払い実績 |

## 表示例

```
【精算】
たろう → 共有口座 へ 45,000円
はなこ → 共有口座 へ 67,000円
```

```
【払い過ぎの場合】
たろう ← 共有口座 から 10,000円（来月分から調整）
はなこ → 共有口座 へ 122,000円
```

## 結果

### ポジティブな影響

- 実際の振込操作と1対1に対応する直感的な表示になった
- `transfer_a + transfer_b = 0` により精算結果の整合性を常に確認できる
- 「払い過ぎ」も同じモデルで表現でき、特殊ケースの分岐が不要

### ネガティブな影響・トレードオフ

- 「A から B へ直接支払う」形式には対応しない（今回の用途には不要）
- 「払い過ぎ」の場合の表示（「来月から調整」）は UX で別途案内が必要
