# ADR 0003: デプロイ先に GitHub Pages を選定する

- **ステータス**: Accepted
- **作成日**: 2026-03-02
- **決定者**: ni0911

## 背景

フロントエンドの静的ファイルをホスティングするサービスを選定する必要があった。
今回のアプリはサーバーサイド処理が不要な SPA（静的ファイルのみ）であり、
個人開発・プライベートリポジトリという前提条件がある。

重視した点:

- 無料で使えること
- プライベートリポジトリでも利用できること
- CI/CD と連携してプッシュ時に自動デプロイできること

## 検討した選択肢

### 選択肢 1: Vercel

- メリット: デプロイが非常に簡単（リポジトリ連携するだけ）。プレビューデプロイ機能が便利。Next.js との相性が抜群。
- デメリット: 無料プランはプライベートリポジトリに制限がある場合がある。Next.js を使わない場合のメリットが薄い。

### 選択肢 2: Netlify

- メリット: 静的サイトホスティングの老舗で信頼性が高い。フォーム処理・関数なども使える。
- デメリット: 無料枠に帯域制限（100GB/月）がある。今回の用途には機能が豊富すぎる。

### 選択肢 3: Firebase Hosting

- メリット: Google インフラで安定。将来 Firestore / Firebase Auth を使う場合にまとめられる。
- デメリット: 将来のデータ層は Supabase を検討しているため Firebase との統合メリットがない。設定がやや複雑。

### 選択肢 4: GitHub Pages（採用）

- メリット: ソースコードと同じ GitHub リポジトリで管理できる。GitHub Actions との連携が公式サポートされており設定が簡単。プライベートリポジトリでも無料で利用できる。追加サービスへのアカウント登録が不要。
- デメリット: SPA のディープリンクに対応していないため、BrowserRouter が使えず HashRouter を採用する必要がある。カスタムドメイン設定が他サービスより手間がかかる。

## 決定

**GitHub Pages を採用する。**

理由: 追加サービスの登録なしに GitHub リポジトリと一体で管理でき、
GitHub Actions による自動デプロイも公式サポートで設定が容易だったため。
個人利用・プライベートリポジトリ・無料という条件をすべて満たす最もシンプルな選択肢。

## 結果

### ポジティブな影響

- `main` ブランチへの push だけで自動デプロイが完了する
- GitHub のリポジトリページからデプロイ状況を一元管理できる
- 追加の外部サービスアカウントが不要

### ネガティブな影響・トレードオフ

- SPA のルーティング制約から HashRouter を採用することになり、URL に `#` が入る（例: `https://ni0911.github.io/waritaro/#/sheet/2026-03`）
- カスタムドメインを設定する場合は別途 DNS 設定が必要

### 今後の対応

- 将来 Supabase を導入してバックエンドが必要になっても、フロントエンドのホスティングは GitHub Pages のままで問題ない
- アクセス数が増えてきた場合は Vercel / Cloudflare Pages への移行を検討する
