# ADR 0002: フロントエンド技術スタックの選定

- **ステータス**: Superseded by [ADR 0005](0005-rails-fullstack-migration.md)
- **作成日**: 2026-03-02
- **決定者**: ni0911

## 背景

同棲2人向けの支払い管理Webアプリを個人開発するにあたり、フロントエンドの技術スタックを選定する必要があった。
要件として以下を重視した。

- TypeScript で型安全に開発できること
- スマホでの使い勝手が良いUI（モバイルファースト）を効率よく作れること
- ビルド後の成果物を静的ファイルとしてデプロイできること（サーバー不要）
- 自分が使い慣れた技術であること

## 検討した選択肢

### 選択肢 1: Next.js + Tailwind CSS

- メリット: ファイルベースルーティングが便利。SSR・SSG・APIルートも使える。Vercel へのデプロイが非常に簡単。
- デメリット: 今回はサーバーサイド機能（SSR / API Routes）が一切不要。`output: export` の静的エクスポートは可能だが、本来の使い方ではない。オーバースペック。

### 選択肢 2: Vue + Vite + Tailwind CSS

- メリット: Vite によるビルドが高速。Vue 3 の Composition API は TypeScript との相性が良い。
- デメリット: 自分のメイン技術スタックが React であり、習熟度の差がある。

### 選択肢 3: React + TypeScript + Vite + Tailwind CSS（採用）

- メリット: Vite により開発サーバー起動・HMRが高速。React は自分の得意技術で開発速度が高い。Tailwind CSS でスマホUIを効率的に構築できる。静的ビルドのみで完結する。
- デメリット: React 単体にはルーティングが含まれないため `react-router-dom` を別途導入する必要がある。

## 決定

**React + TypeScript + Vite + Tailwind CSS v3 を採用する。**

理由: 自分が最も習熟している技術スタックであり、個人開発の速度を最大化できる。
今回の要件（静的ファイルとしてデプロイ・モバイルUI）に対して必要十分な構成であり、
Next.js のようなサーバーサイド機能は不要なため Vite ベースで十分と判断した。

## 結果

### ポジティブな影響

- Vite の高速 HMR により開発体験が良好
- Tailwind CSS v3 でモバイルファーストの UI を素早く構築できた
- TypeScript による型安全な開発で実装ミスを早期に検出できた

### ネガティブな影響・トレードオフ

- ルーティングに `react-router-dom` を別途導入する必要があった
- Next.js のような規約ベースの構造がないため、ディレクトリ設計を自分で決める必要があった

### 今後の対応

- Supabase 移行後もこのスタック構成は維持する予定
- より複雑なサーバーサイド処理が必要になった場合は Next.js への移行を検討する
