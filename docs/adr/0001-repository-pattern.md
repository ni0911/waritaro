# ADR 0001: データアクセス層にリポジトリパターンを採用する

- **ステータス**: Superseded by [ADR 0005](0005-rails-fullstack-migration.md)
- **作成日**: 2026-03-02
- **決定者**: ni0911

## 背景

同棲支払い管理アプリ「ワリタロ会計部」の初期リリースでは、デプロイコストを抑えるために
LocalStorage をデータ保存先として採用した。
しかし将来的に複数端末での同期が必要になった際、Supabase へ移行する可能性が高い。

この移行を最小コストで行うために、データアクセス層の設計を初期段階から抽象化しておく必要があった。

## 検討した選択肢

### 選択肢 1: 直接 localStorage 操作（パターンなし）

各 hooks やコンポーネントから `localStorage.getItem / setItem` を直接呼ぶシンプルな実装。

- メリット: コード量が最小、学習コストゼロ
- デメリット: Supabase 移行時に呼び出し箇所を全て書き換える必要がある。テストも困難。

### 選択肢 2: リポジトリパターン（インターフェース＋実装分離）

`ICardRepository` などのインターフェースを定義し、`LocalStorageCardRepository` が実装する。
`src/repository/index.ts` でインスタンスを生成・エクスポートする。

- メリット: `index.ts` の import を差し替えるだけで Supabase 移行が完了する。hooks / services の変更不要。
- デメリット: インターフェース・実装・DI 層の三層構造でコード量が増える。

### 選択肢 3: Zustand / Jotai などの状態管理ライブラリ

グローバルストアでデータを管理し、永続化プラグインで LocalStorage に保存する。

- メリット: React エコシステムと親和性が高い
- デメリット: ライブラリへの依存が増える。Supabase 移行時はストア設計ごと見直しが必要になる可能性がある。

### 選択肢 4: React Query + fetch

サーバー状態管理ライブラリを使い、将来の API 呼び出しを想定した設計にする。

- メリット: キャッシュ・再フェッチ・ローディング状態管理が充実している
- デメリット: LocalStorage のみで動く初期段階には明らかにオーバースペック。

## 決定

**選択肢 2（リポジトリパターン）を採用する。**

理由: Supabase 移行コストを最小化することが最優先だったため。
`src/repository/index.ts` 1ファイルの import を差し替えるだけで移行が完了する設計は、
将来の運用コストを大幅に削減できると判断した。

## 結果

### ポジティブな影響

- Supabase 移行時の変更箇所が `src/repository/index.ts` の 4 行のみで済む
- hooks・services・pages はデータソースを意識しない設計になった
- 将来モックリポジトリを差し込むことでテストが容易になる

### ネガティブな影響・トレードオフ

- インターフェース / LocalStorage 実装 / DI エントリーポイントの三層構造により、初期コード量が増加した
- 個人・小規模アプリとしては設計が重い（過剰文漫になる可能性がある）
- LocalStorage は本来同期 API だが、Supabase 移行を見据えて全メソッドを `async/await` に統一したため、実装が若干複雑になった

### 今後の対応

- Supabase 移行時は `src/repository/supabase/` に同インターフェースの実装を作成し、`src/repository/index.ts` の import を差し替えるだけで完了する
- ID 生成は `crypto.randomUUID()`、日時は ISO8601 で統一済みのため、移行時のデータ形式の変換は不要
